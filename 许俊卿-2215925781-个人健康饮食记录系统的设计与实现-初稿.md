\[摘 要\]随着居民生活水平不断提高，高热量饮食和久坐少动等生活方式变得日益普遍，超重、肥胖及相关慢性疾病的风险也在持续上升，这使得健康饮食管理的需求越来越突出。然而，传统的手工记录方式在实际使用中存在不少问题，比如操作比较繁琐、数据分析能力有限，而且用户往往很难长期坚持下来。针对这些不足，本文设计并实现了一套基于前后端分离架构的个人健康饮食记录系统。系统主要由UniApp用户端、Vue管理端、Flask后端和SQLite数据库组成，涵盖了用户信息管理、饮食记录、食物查询、营养统计分析以及AI饮食建议等功能。管理端则提供了数据概览、用户与食物管理、系统配置等模块，方便日常维护。为了提升饮食数据分析的准确性和可解释性，系统加入了身体状态快照机制，将每次饮食记录与用户当时的体重、体脂等信息关联起来，避免了因身体状态变化导致的分析偏差。此外，系统还结合大语言模型，能够根据用户的饮食数据和身体状态生成个性化的饮食建议，并提供智能问答服务。经过测试，系统功能比较完善，运行也较为稳定，具备良好的实用价值和一定的扩展性。

Design and Implementation of a Personal Healthy Diet Recording System

Software Engineering Major XU Jun-qing

**Abstract:**As living standards continue to rise, high-calorie diets and sedentary lifestyles have become increasingly common, leading to a steady rise in the risk of overweight, obesity, and related chronic diseases. This has made the need for healthy dietary management increasingly urgent. However, traditional manual recording methods face numerous practical challenges, such as cumbersome operations, limited data analysis capabilities, and the difficulty users often face in maintaining consistency over the long term. To address these shortcomings, this paper designs and implements a personal healthy diet tracking system based on a frontend-backend separation architecture. The system primarily consists of a UniApp client, a Vue admin panel, a Flask backend, and an SQLite database, covering functions such as user information management, diet logging, food lookup, nutritional statistics and analysis, and AI-powered dietary recommendations. The admin panel provides modules for data overviews, user and food management, and system configuration to facilitate daily maintenance. To enhance the accuracy and interpretability of dietary data analysis, the system incorporates a body status snapshot mechanism that links each dietary entry to the user's current weight, body fat percentage, and other relevant information, thereby preventing analytical biases caused by changes in physical condition. Additionally, the system integrates a large language model to generate personalized dietary recommendations based on the user's dietary data and physical condition, while also providing intelligent Q&A services. Testing has shown that the system's functionality is comprehensive, its operation is stable, and it offers good practical value and a degree of scalability.

**Keywords:**diet tracking; personalized recommendation; vue; flask

# 绪论

## 研究背景

社会经济的快速发展和居民生活水平的提高，人们的饮食结构发生了明显变化。高热量、高脂肪、高糖分食物的摄入日益增加，久坐少动的生活方式也逐渐普遍，导致超重、肥胖以及糖尿病、高血压等慢性疾病风险持续上升<sup>\[1\]</sup>。在这样的现实背景下，如何通过信息化手段帮助用户进行饮食自我管理，逐渐成为健康管理领域的重要研究方向。

随着移动互联网、前后端分离技术的发展，个人健康管理系统有了较好的实现基础。移动端能够使饮食记录更加方便，Web管理端能够方便管理平台、治理平台数据，后端服务可以完成登录、营养计算、统计、分析以及与AI对接等一系列功能。大语言模型的发展也为饮食建议和健康问答提供了新的交互方式，使系统不仅能够完成数据记录，还能够进一步提供自然语言形式的辅助反馈<sup>\[2\]</sup>。

## 国内外研究现状

国外在健康饮食管理应用方面起步较早，MyFitnessPal、Yazio、Lifesum等应用已经积累了较丰富的食物数据库和成熟的记录分析能力。相关系统通常能够提供热量追踪、营养素统计、目标设定、体重变化分析以及一定程度的个性化建议。近年来，随着人工智能技术的发展，一些研究开始尝试将图像识别、自然语言问答和个性化推荐引入饮食管理领域<sup>\[3\]</sup>。

国内在健康管理和移动健康应用方面也取得了较快发展。薄荷健康、Keep、华为健康等产品已经在用户饮食记录、运动协同和体重管理方面积累了较多经验。与此同时，国内研究在基于《中国食物成分表》的营养数据库建设、个性化目标热量计算、健康管理系统设计等方向也有较多成果<sup>\[4\]</sup>。

基于以上分析，本文设计实现的个人健康饮食记录系统，尝试在用户端、管理端、营养计算和AI辅助建议之间形成较为完整的闭环，从而增强系统的实用性和完整性。

## 研究目的和意义

本课题的研究目的在于设计并实现一套个人健康饮食记录系统，使用户能够方便地完成个人饮食数据记录，并基于身体状态、健康目标和营养数据获得更直观、更具个性化的分析结果与辅助建议。系统不仅关注"记录了什么"，还关注"记录结果如何解释"以及"如何帮助用户形成更健康的饮食习惯"。

从研究目的上看，本课题主要希望实现以下目标：构建一套完整的饮食记录业务闭环，包括用户注册登录、身体档案维护、食物库查询、饮食记录新增与编辑、统计分析和后台管理等功能模块。基于用户的身高、体重、生日、性别、活动水平和健康目标，计算其BMI、BMR以及每日推荐热量摄入，使饮食分析能够与个体状态紧密结合。通过食物成分数据、饮食记录和统计分析，帮助用户更直观地理解自身饮食结构。引入AI辅助能力，实现饮食建议、饮食评价和智能问答，提高系统交互体验。提供后台管理功能，为食物库维护、用户管理、系统配置和日志审计提供支持。

## 研究主要内容

我们所研发的面向个人健康管理的智能饮食记录与分析平台基于前后端分离架构设计实现，用户管理、饮食记录、统计分析、AI辅助、后台管理功能均可使用。平台综合考虑用户的个人身体状况、健康需求及历史信息对饮食结构进行分析，并得出营养评价和营养建议，为个体健康管理提供参考。

在系统实现方面，后端基于Flask框架，完成了用户认证、健康档案管理、食物库查询、饮食记录及营养分析等功能。系统可根据用户录入的食物种类和摄入量，自动计算热量及三大营养素（蛋白质、脂肪、碳水化合物），并结合用户身高、体重、年龄等信息计算BMI、BMR及每日推荐摄入量，对用户当日及阶段性饮食情况进行统计分析并生成可视化结果。

# 相关技术

## Flask框架

Flask是Python生态中应用较为广泛的轻量级Web框架，具有结构简洁、扩展灵活、上手成本低等特点<sup>\[5\]</sup>。与Django、Spring Boot等大型一体化框架相比，Flask并不预设固定的项目结构，也不强制引入ORM、模板引擎或用户认证等内置模块，而是将技术选型的主动权交还给开发者。开发者可以依据实际业务需要，自由选择数据库访问、身份认证、跨域处理、参数校验等配套组件，按需组装，避免引入与项目无关的冗余依赖，使各模块之间的职责边界保持清晰。

对于毕业设计项目或中小型信息管理系统而言，这一特点尤为契合。此类系统的业务逻辑通常并不复杂，开发重心更多在于需求梳理、接口设计与前后端协调，而非框架本身的学习与适配。Flask的核心API精简直观，开发者在掌握基本概念后便可较快进入实际开发阶段，整体学习曲线相对平缓。因此，综合开发效率、项目规模与可维护性等因素考量，Flask对于本系统的后端建设具有较强的适用性。

在本系统中，Flask主要承担后端核心业务服务的实现任务，负责处理用户认证、健康档案维护、食物管理、饮食记录、统计分析、AI接口调用以及后台管理等业务逻辑。同时，Flask能够较方便地与SQLAlchemy、JWT、CORS等扩展组件集成，满足本系统在数据库操作、身份验证和前后端分离通信方面的需求<sup>\[6\]</sup>。综合来看，Flask的轻量化特征与本项目以营养计算和统计分析为核心、而非高并发事务处理的业务特点相契合——相比Django等重型框架能减少不必要的内置抽象层，使开发者更聚焦于业务逻辑本身；相比FastAPI等异步框架，Flask的学习曲线更平滑，且本项目的核心业务（营养计算、AI调用）在请求量可控的情况下同步模式已足够胜任。因此，Flask成为本项目后端实现的主要技术基础。

## SQLite数据库

SQLite是一种典型的轻量级关系型数据库管理系统，具有零配置、占用资源少、部署简单和可移植性强等优点<sup>\[7\]</sup>。与MySQL、PostgreSQL等需要独立部署数据库服务的系统不同，SQLite将整个数据库直接存储为单一文件，开发者无需额外安装和维护数据库服务器即可完成数据持久化操作。这种特性使其非常适合课程设计、原型系统、中小型管理系统以及开发测试环境使用，能够有效降低系统搭建和运行门槛。

对于个人健康饮食记录系统而言，系统当前阶段的数据规模和并发量相对可控，核心数据主要包括用户信息、身体档案、食物库、饮食记录、系统配置和AI调用记录等内容，采用SQLite完全能够满足项目实现需求。借助关系型数据库中的主键、外键和结构化查询机制，系统可以较好地维护用户、身体快照、食物信息与饮食记录之间的数据关联关系，从而保证统计分析结果的准确性与一致性。此外，SQLite与SQLAlchemy ORM结合后，可以进一步简化数据模型设计和数据库访问逻辑，提高后端开发效率<sup>\[8\]</sup>。在部署层面，SQLite的零配置、单文件存储特性消除了独立数据库服务的依赖，降低了评审和演示时的环境搭建成本。后续若因用户规模增长需要迁移至MySQL或PostgreSQL，借助SQLAlchemy ORM仅需修改数据库连接配置即可完成切换，无需重写数据访问代码。因此，在本系统中选用SQLite，不仅符合项目实际规模，也兼顾了实现成本、维护便利性和功能完整性，同时保留了后续扩展的灵活性。

## Vue3与uni-app技术

Vue3是当前前端开发中应用广泛的渐进式JavaScript框架，具备组件化开发、响应式数据绑定、组合式API和良好工程化支持等特点，能够有效提升复杂界面的开发效率与代码可维护性<sup>\[9\]</sup>。uni-app则是在Vue语法基础上发展起来的跨端开发框架，支持开发者通过一套代码同时面向多种移动端运行环境进行构建<sup>\[10\]</sup>，适合需要兼顾开发效率与多端适配能力的应用场景。两者都以组件化思想为核心，能够将页面拆分为结构清晰、职责明确的独立模块，从而降低前端系统的复杂度。

本系统在前端采用"双端分离"的实现方式，其中用户端基于uni-app开发，主要面向日常饮食记录和健康数据查看场景；管理端基于Vue3开发，主要面向后台数据治理与系统维护场景。用户端更强调移动端交互便捷性，需要支持用户随时进行饮食录入、食物查询、统计查看和AI问答，因此uni-app的跨端与移动适配优势能够较好满足系统需求。管理端则更关注数据列表展示、表单维护、权限操作和配置管理，Vue3结合现代前端构建工具和组件库，能够快速实现结构清晰、交互规范的后台管理界面。选择uni-app而非Flutter或原生开发，主要出于两点考虑：一是uni-app基于Vue语法，与Vue3管理端共享相同的组件化开发模式和响应式数据绑定理念，降低了全栈开发的学习成本；二是本项目需同时覆盖微信小程序和H5两端，uni-app的跨端编译能力能直接满足这一需求，避免为不同平台维护多套代码。通过同时引入Vue3与uni-app，本系统既具备移动端便捷记录能力，又能满足后台管理需求，从而形成较完整的前端应用体系<sup>\[11\]</sup>。

## JWT身份认证技术

JWT（JSON Web Token）是一种基于JSON格式进行信息安全传输的身份认证技术，广泛应用于前后端分离架构中<sup>\[12\]</sup>。其核心思想是在用户登录成功后，由服务器根据用户身份信息生成一个具备签名校验能力的令牌，并将其返回给客户端；客户端在后续访问受保护资源时，只需在请求头中携带该令牌，服务器即可通过解析与校验完成身份确认，而不必在服务端额外保存传统会话状态。相较于基于Session的认证方式，JWT更适合接口化、跨端化的系统架构，能够降低服务端会话维护成本并提升系统扩展灵活性<sup>\[13\]</sup>。

在个人健康饮食记录系统中，JWT技术主要用于实现普通用户端和管理端的统一认证控制。系统在用户或管理员登录成功后生成对应类型的令牌，客户端在调用需要授权的接口时携带该令牌，后端再通过装饰器或中间层逻辑校验其合法性，并依据角色信息决定是否允许访问目标资源。由于本系统同时包含用户端和后台管理端，且两类角色的权限范围存在明显差异，因此引入JWT不仅能够满足基本登录状态保持需求，还能为用户权限控制、管理员权限划分以及敏感接口保护提供技术支撑<sup>\[14\]</sup>。该技术的应用使系统身份认证机制更加清晰，也更符合现代Web应用的实现方式。

## 饮食记录与食物管理模块详细设计

饮食记录与食物管理模块是本系统的核心业务模块。用户所有的营养统计、饮食分析和AI辅助结果，最终都建立在饮食记录数据和标准化食物数据之上。因此，本模块设计的重点不只是"记录一顿饭"，而是通过标准食物库、统一营养换算规则和结构化餐次组织方式，为后续分析提供稳定的数据基础。饮食记录与食物管理模块总体流程图如图5-1所示。

### 饮食记录添加与营养换算设计

饮食记录新增流程由记录页（record.vue）与食物选择页（foods.vue）共同完成。用户在记录页选择日期与餐次（早餐、午餐、晚餐、加餐四类），点击"添加食物"后进入食物选择页，并以选择模式浏览食物列表。

在食物页中，用户选定食物后输入重量，并将其添加到对应日期与餐次。随后，前端将food_id、date、meal_type和weight_g提交至后端/api/meals接口。

后端在server/app/api/meal.py中处理该请求，主要进行餐次合法性、重量有效性、食物存在性及审核状态校验。校验通过后，根据食物每100克营养成分与实际重量计算热量、蛋白质、脂肪和碳水，并写入MealRecord。

在营养计算前，系统会调用get_or_create_body_record检查当天是否存在身体快照；若不存在，则基于用户身高、体重、BMI、BMR及营养比例生成当日快照，并与本次饮食记录绑定，从而保证饮食数据与身体状态的一致性。饮食记录添加与营养换算时序图如图5-2所示。

### 饮食记录查询、编辑与按餐次组织设计

记记录页同时承担新增与日常管理功能。页面顶部支持按日期切换，用户可按天查看历史饮食记录；中部通过早餐、午餐、晚餐和加餐四个标签切换餐次；下方展示当前餐次的食物列表及热量、蛋白质、脂肪和碳水等营养小计。该设计帮助用户形成"按天、按餐次"的记录习惯，提升使用清晰度与操作效率。

后端通过 /api/meals 的 GET 接口按用户与日期查询记录，并按 meal_type 分组，同时计算当日各类营养总计后返回前端。前端基于返回数据生成餐次列表与统计信息，实现快速切换与局部刷新展示。

在记录维护方面，系统支持编辑与删除操作。用户可在前端进入编辑界面修改食物重量，系统通过 /api/meals/{id} 更新数据并重新计算营养值；删除操作同样通过该接口完成，并在前端增加确认提示以避免误操作。

### 每日营养统计与评分设计

系统的每日统计主要围绕当天所有饮食记录进行汇总。后端通过server/app/api/stats.py中的/api/stats/today接口，先汇总当天热量、蛋白质、脂肪和碳水的实际摄入值，再结合用户当前的每日热量目标与营养素比例计算目标值、剩余值和摄入状态。与此同时，系统还会根据实际摄入与目标之间的差距计算饮食评分，并返回BMI及BMI状态等辅助解释信息。

在前端用户端中，首页和统计页都会使用这部分数据。统计页可以展示已摄入热量与目标热量对比、剩余可摄入值以及摄入状态，从而让用户快速了解当天是否存在超标或不足。相比简单地呈现总热量数字，这种设计更符合健康管理场景下"记录之后要有解释"的系统目标。

系统的营养计算均集中在NutritionService服务类中统一实现，核心算法如下：（1）BMI采用国际通用公式BMI = 体重(kg) / 身高(m)²，状态判定采用中国标准（偏瘦<18.5、正常18.5~24、超重24~28、肥胖≥28）；（2）基础代谢率BMR采用Mifflin-St Jeor公式，男性为10×体重+6.25×身高-5×年龄+5，女性为10×体重+6.25×身高-5×年龄-161；（3）每日推荐热量采用TDEE法计算，即BMR × 活动系数（久坐1.2/轻度1.375/中度1.55/高强度1.725）× 目标系数（维持1.0/减脂0.85/增肌1.15）；（4）营养素目标值按热量分配计算，蛋白质(g) = 日热量×蛋白质占比÷4，脂肪(g) = 日热量×脂肪占比÷9，碳水(g) = 日热量×碳水占比÷4；（5）饮食评分采用加权评分法，热量维度权重40%，蛋白质、脂肪、碳水维度各占20%，各维度以实际摄入量与目标值的比值计算单项得分（80%~120%区间内为满分，超出该区间按偏离比例线性扣减），加权汇总后得到0~100分的综合评分。每日营养统计与评分时序图如图5-4所示。

### AI饮食建议与饮食评价设计

AI饮食建议与饮食评价功能位于 server/app/api/ai.py 中。系统在调用模型前，会先整合用户基础健康信息、当日饮食摄入数据及目标值等上下文，而不是直接将用户问题发送给AI，从而使生成结果更贴合实际饮食场景。

在饮食建议功能中，后端先统计用户当天的热量及三大营养素摄入情况，并与每日目标进行对比，计算剩余摄入空间，再生成简要建议并交由AI输出自然语言结果。饮食评价功能则在此基础上增加饮食评分，并结合目标差异生成评价上下文，使AI能够给出总结性反馈。前端AI页面中，用户点击"今日饮食评价"或"饮食建议"卡片即可触发请求，返回结果会直接展示在对话区域。

该设计的优势在于：既保留了数据统计的客观性，又通过自然语言输出提升了用户的理解体验，使非专业用户也能直观获取饮食分析结果。

在异常处理方面，AI模块设计了多层次的兜底策略：当API密钥未配置时，系统提示"AI功能暂未开启"；当请求超时（30秒阈值）或上游服务返回错误状态码时，系统提取上游错误信息并以友好提示返回前端；当模型返回空内容时，给出明确的错误反馈而非静默失败。此外，系统为每位用户设置了每日AI调用次数上限，超过限额时提示"今日调用次数已用完"，避免因异常调用导致成本失控。每次AI调用均记录其调用类型、提示词、返回内容、模型名称、Token消耗和响应时长，保存为AISuggestion历史记录，便于管理端进行调用统计和问题追溯。AI饮食建议与饮食评价时序图如图5-5所示。

### 用户管理设计

后台用户管理页面位于 client/admin-web/src/views/users/UsersPage.vue，支持关键词搜索、冻结状态筛选、分页查看及用户冻结/解冻操作。整体采用"列表 + 详情抽屉"结构：列表用于快速定位用户，抽屉用于展示BMI、目标热量、身体信息及饮食偏好等详细数据，在效率与信息完整性之间取得平衡。

### 食物与分类管理设计

食物与分类管理是后台数据治理的核心模块之一。食物管理页面 FoodsPage.vue 支持按关键字、分类及审核状态筛选食物，并提供新增、编辑、删除及审核通过等操作。页面中的食物信息、营养成分与审核状态等字段均与后端数据直接映射，可直观反映当前食物库状态。

后端食物管理接口位于 server/app/api/admin/foods.py，管理员可通过接口实现食物的查询、新增、修改、删除及审核操作（如 /api/admin/foods/{id}/approve）。系统在关键操作时同步记录管理员日志，以确保操作可追溯。

分类管理页面 CategoriesPage.vue 用于维护 FoodCategory 分类结构。左侧展示分类列表并支持编辑，右侧根据 parent_code 生成树形结构，便于检查分类层级关系。后端通过 /api/admin/categories 提供分类的增删改查功能。该设计保证了前台分类检索的稳定性，同时降低了后台维护错误率。

从面向对象设计的角度看，系统的核心类及其关系如下：User（用户）与BodyRecord（身体快照）为一对多聚合关系——用户表维护当前身体状态，BodyRecord表保存各时间点的快照；User与MealRecord（饮食记录）为一对多关联；每条MealRecord同时关联一个Food（食物）和一个BodyRecord，形成"谁在什么身体状态下吃了什么食物"的完整记录链；NutritionService作为无状态服务类，以静态方法提供BMI计算、BMR计算、营养素目标计算、饮食评分计算和摄入状态判定等功能；AIService封装大语言模型调用逻辑，提供上下文拼接、流式输出、调用次数管理和异常兜底等方法。该设计将数据模型与业务逻辑解耦，各服务类可独立测试和替换。

系统首页和统计分析页都会调用今日统计接口获取当日营养汇总结果。后端get_today_stats()方法会先读取用户当天全部饮食记录，再结合用户当前热量目标与营养素比例计算目标值、剩余值、按餐次汇总结果以及饮食评分。前端则据此展示今日已摄入热量、剩余热量、BMI状态、营养素完成情况和餐次分布结果。

上述接口中调用的饮食评分算法calculate_diet_score()是系统核心业务逻辑之一，其关键实现如下：

def calculate_diet_score(actual, target):
    scores = []
    # 热量评分 (权重40%)
    if target['calorie'] > 0:
        calorie_ratio = actual['calorie'] / target['calorie']
        if 0.9 <= calorie_ratio <= 1.1:
            calorie_score = 100
        elif 0.8 <= calorie_ratio <= 1.2:
            calorie_score = 80
        else:
            calorie_score = max(0, 100 - abs(calorie_ratio - 1) * 100)
    scores.append(('calorie', calorie_score, 0.4))
    # 三大营养素评分 (各权重20%)
    for nutrient in ['protein', 'carb', 'fat']:
        if target[nutrient] > 0:
            ratio = actual[nutrient] / target[nutrient]
            if 0.8 <= ratio <= 1.2:
                score = 100
            else:
                score = max(0, 100 - abs(ratio - 1) * 80)
        scores.append((nutrient, score, 0.2))
    # 加权总分
    total = sum(score * weight for _, score, weight in scores)
    return round(total)

该评分算法将热量（40%权重）与三大营养素（各20%权重）的实际摄入与目标值的偏离程度量化为0~100分的综合评分，在80%~120%的目标区间内视为满分，超出部分按比例线性扣减，最终通过加权求和得到当日饮食平衡度的量化评价。

综上所述，本章通过对系统主要功能模块的测试，验证了个人健康饮食记录系统在当前实现条件下能够较好地完成预期业务目标，也为后续系统优化与完善提供了实践依据。

在非功能性验证方面，经手工测试，核心接口（食物搜索、饮食记录查询、今日统计）在包含多条数据的场景下响应流畅，无明显卡顿；AI接口在正常网络条件下的响应时间在3至10秒范围内，超时场景下能够正确返回友好提示；异常输入（空值、非法值、超长文本）均被前后端校验机制拦截并给出明确提示。需要指出的是，当前测试以功能联调和人工验证为主，尚未建立自动化测试体系，后续可补充接口自动化测试和压力测试以进一步验证非功能需求指标的达成情况。

结束语

本文围绕个人健康饮食记录系统的设计与实现，完成了从需求分析、系统设计、编码实现到功能测试的全过程。系统采用前后端分离架构，以UniApp用户端、Vue管理端、Flask后端和SQLite数据库为主要技术基础，实现了用户注册登录、身体档案维护、饮食记录、食物库查询、营养统计分析、AI建议与问答以及后台管理等主要功能。
