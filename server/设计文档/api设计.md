以下是基于你提供的 **模块设计** 与 **数据库设计** 所整理的 **后端 API 设计（RESTful 风格）**，遵循以下原则：

- **简洁实用**：每个接口职责单一、命名清晰；
- **安全规范**：所有用户数据操作需鉴权（JWT 或 Session）；
- **无外键依赖**：符合你的“无耦合”数据库规范；
- **快照一致性**：饮食记录绑定 `body_record_id`，确保热量目标与身体状态一致；
- **轻量 AI**：AI 建议作为可选功能，带限流和开关控制。

## 👤 用户 & 身体档案

| 方法 | 路径                 | 描述                                                         |
| ---- | -------------------- | ------------------------------------------------------------ |
| GET  | `/user/profile`      | 获取当前用户基本信息 + 当前身体档案（含 BMI、BMR、热量目标） |
| PUT  | `/user/profile`      | 更新身体档案（身高、体重、活动水平等）→ **自动创建新 body_records 快照** |
| GET  | `/user/body-history` | 获取身体记录历史（按时间倒序，用于趋势分析）                 |

> 💡 每次更新身体档案时，后端：
> 1. 用 `birthday` 精确计算当前年龄；
> 2. 用 Mifflin-St Jeor 公式计算 BMR；
> 3. 结合活动水平算出 `daily_calorie_goal`；
> 4. 插入一条新 `body_records`；
> 5. 后续饮食记录将引用此 `body_record_id`。

## 🍽️ 饮食记录

| 方法   | 路径                     | 描述                                                         |
| ------ | ------------------------ | ------------------------------------------------------------ |
| POST   | `/meals`                 | 新增一条饮食记录**必传字段**：`date`, `meal_type`, `food_id`, `weight_g`→ 后端自动关联 **当日最新 body_record_id** |
| GET    | `/meals?date=2026-01-24` | 获取指定日期所有餐次记录（含食物详情、营养值）               |
| PUT    | `/meals/{recordId}`      | 修改某条记录（可改重量、食物、餐次）                         |
| DELETE | `/meals/{recordId}`      | 删除单条记录（软删除或物理删均可）                           |

> ⚠️ **关键逻辑**：新增记录时，后端需查询该用户在 `date` 当天或最近一次的 `body_records`，取其 `id` 作为 `body_record_id` 存入 `meal_records`。

## ✅ 补充说明

- **错误响应统一格式**：
  ```json
  { "error": "INVALID_TOKEN", "message": "登录已过期" }
  ```
- **分页**：列表接口支持 `page=1&size=20`
- **时间格式**：全部使用 `YYYY-MM-DD`（日期）或 ISO8601（时间戳）
- **单位统一**：重量为克（g），热量为 kcal，营养素为 g
