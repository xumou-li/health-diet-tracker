6  系统编码与实现	
6.1  用户认证模块实现
用户认证与健康档案模块是整个系统的基础模块。用户只有在完成注册和登录后，才能继续进行饮食记录、营养分析和AI辅助操作；而健康档案中的性别、年龄、身高、体重、活动水平和健康目标等信息，又直接参与BMI、BMR、每日推荐热量和营养目标的计算。因此，该模块不仅承担身份认证功能，同时也是系统后续分析能力的数据基础。
6.1.1  用户注册功能实现
用系统用户端登录与注册页面位于client/uni-app-user/src/pages/login/login.vue。该页面采用登录与注册同页切换的实现方式，并在注册流程中设计了两步式表单：第一步录入手机号与密码，第二步录入性别、出生日期、身高、体重、活动水平、健康目标、热量系数以及营养素比例。这样设计可以有效减少移动端单屏输入压力，同时让用户在注册阶段就完成身体档案初始化。
在注册过程中，前端会先校验密码确认结果，并依据出生日期、性别、体重和活动水平等信息调用本地营养计算工具，对预计每日热量进行预估显示。后端注册接口位于server/app/api/auth.py的register()方法中。接口在接收到注册请求后，会先校验手机号、邮箱、密码格式及身体参数范围，然后计算年龄、BMI、BMR、每日推荐热量和营养素比例，最后将结果写入User和BodyRecord表中。由此可见，系统注册不是简单的账号创建过程，而是把账号信息和健康基础档案一并初始化完成。
用户登录时，需要输入手机号或邮箱以及密码。后端login()接口会自动判断当前输入属于手机号还是邮箱，并据此查找用户信息；若账号被冻结，系统会直接返回拦截提示；若校验成功，则统一签发JWT令牌返回前端。前端在登录成功后保存令牌，并在后续请求中自动携带，实现用户态的统一鉴权控制。其关键代码如下：
@auth_bp.route('/login', methods=['POST'])
def login():
    data = request.get_json() or {}
    account = data.get('account')
    password = data.get('password')

    if validate_phone(account):
        user = User.query.filter_by(phone=account, is_deleted=False).first()
    elif validate_email(account):
        user = User.query.filter_by(email=account, is_deleted=False).first()
    else:
        user = User.query.filter(
            (User.phone == account) | (User.email == account),
            User.is_deleted == False
        ).first()

    if not user:
        return error('账号不存在')
    if user.is_frozen:
        return error('账号已被冻结，请联系管理员')
    if not check_password_hash(user.password_hash, password):
        return error('密码错误')

    access_token = create_access_token(
        identity=json.dumps({'type': 'user', 'id': user.id})
    )
return success({'token': access_token, 'user': user.to_dict()}, '登录成功')

6.1.2  个人健康档案维护实现
系统个人健康档案维护功能主要由个人中心页面和用户档案接口共同完成。用户端页面能够展示当前BMI、BMR、每日热量目标以及蛋白质、脂肪、碳水化合物比例等信息；当用户修改身高、体重、活动水平、健康目标或热量系数后，后端会重新进行营养指标计算，并把更新结果保存到用户主表中。这样设计使得健康目标不再只是界面参数，而是能够真正影响后续饮食统计与分析结果的业务数据。
后端档案查询接口位于server/app/api/user.py的get_profile()方法中。系统在获取档案时，不仅返回基础用户信息，还会额外返回BMI状态、营养目标值以及最近一次身体快照，使前端可以直接完成综合展示，而不必自行拼装多张表的数据。关键代码如下：
@user_bp.route('/profile', methods=['GET'])
@login_required
def get_profile():
    user = g.current_user
    latest_body = BodyRecord.query.filter_by(user_id=user.id).order_by(
        BodyRecord.recorded_at.desc()
    ).first()
    default_ratios = NutritionService.get_default_nutrient_ratios(user.health_goal)
    nutrient_targets = NutritionService.calculate_nutrient_targets(
        user.daily_calorie_goal,
        float(user.protein_ratio) if user.protein_ratio is not None else default_ratios['protein'],
        float(user.fat_ratio) if user.fat_ratio is not None else default_ratios['fat'],
        float(user.carb_ratio) if user.carb_ratio is not None else default_ratios['carb']
    )
    data = user.to_dict()
    data['bmi_status'] = NutritionService.get_bmi_status(float(user.bmi))
    data['nutrient_targets'] = nutrient_targets
    data['latest_body_record'] = latest_body.to_dict() if latest_body else None
    return success(data)

6.2  饮食记录与食物管理模块实现
饮食记录与食物管理模块是系统中用户使用最频繁的业务模块。系统并没有采用自由文本输入的方式记录食物，而是要求用户先从食物库中检索和选择食物，再填写摄入重量，由后端统一计算热量和三大营养素结果。这样不仅保证了数据结构化，也为后续统计分析和AI辅助提供了稳定的数据来源。
6.2.1  饮食记录新增与编辑实现
用户进入饮食记录页后，可以先选择日期，再选择早餐、午餐、晚餐或加餐等不同餐次，然后跳转至食物页选择具体食物。前端记录页record.vue中实现了日期切换、餐次切换、记录列表展示、重量编辑和删除确认等交互。用户点击“添加食物”后，页面会携带当前日期与餐次参数跳转到食物库页面，以减少重复输入。
后端饮食记录新增接口位于server/app/api/meal.py的add_meal()方法中。该接口会先校验食物编号、餐次和重量参数，再根据食物每100克的营养值与当前输入重量换算本次摄入结果，并将计算后的热量、蛋白质、碳水和脂肪信息保存到MealRecord表中。与此同时，系统还会自动为该记录绑定当天的身体快照，以确保饮食记录与身体状态保持关联。关键代码如下：
def calculate_nutrition(food, weight_g):
    ratio = float(weight_g) / 100
    return {
        'calorie': int(food.calorie_per_100g * ratio),
        'protein': round(float(food.protein_per_100g) * ratio, 2),
        'carb': round(float(food.carb_per_100g) * ratio, 2),
        'fat': round(float(food.fat_per_100g) * ratio, 2)
    }
@meal_bp.route('/meals', methods=['POST'])
@login_required
def add_meal():
    user = g.current_user
    data = request.get_json() or {}
    food = Food.query.get(data.get('food_id'))
    body_record = get_or_create_body_record(user)
    nutrition = calculate_nutrition(food, data.get('weight_g'))
    meal = MealRecord(
        user_id=user.id,
        food_id=data.get('food_id'),
        body_record_id=body_record.id,
        date=date.fromisoformat(data.get('date', date.today().isoformat())),
        meal_type=data.get('meal_type'),
        weight_g=data.get('weight_g'),
        calorie=nutrition['calorie'],
        protein=nutrition['protein'],
        carb=nutrition['carb'],
        fat=nutrition['fat']
    )
    db.session.add(meal)
    db.session.commit()
    return success(meal.to_dict(), '添加成功')

6.3  统计分析与智能辅助模块实现
统计分析与智能辅助模块是本系统区别于普通饮食记录工具的核心体现。系统不仅能够汇总用户当日热量和营养素摄入情况，还能够进一步实现近7天、近30天的趋势分析、体重变化分析、饮食评分、超标来源分析，以及基于上下文信息的AI饮食建议和智能问答功能。通过该模块，系统能够把原始记录数据转化为更具解释性的健康反馈结果。
6.3.1  每日营养统计与饮食评分实现
系统首页和统计分析页都会调用今日统计接口获取当日营养汇总结果。后端get_today_stats()方法会先读取用户当天全部饮食记录，再结合用户当前热量目标与营养素比例计算目标值、剩余值、按餐次汇总结果以及饮食评分。前端则据此展示今日已摄入热量、剩余热量、BMI状态、营养素完成情况和餐次分布结果。
进一步地，系统还实现了analysis接口，用于输出热量是否超标、超标百分比、超标食物来源、个性化建议和推荐食物等更具解释性的结果。这使得统计页面不再只是简单的数据罗列，而是具备了较明显的健康反馈能力。今日统计的关键代码如下：
@stats_bp.route('/today', methods=['GET'])
@login_required
def get_today_stats():
    user = g.current_user
    today = date.today()
    stats = get_daily_stats(user.id, today)
    targets = NutritionService.calculate_nutrient_targets(
        user.daily_calorie_goal,
        float(user.protein_ratio) if user.protein_ratio else 0.15,
        float(user.fat_ratio) if user.fat_ratio else 0.25,
        float(user.carb_ratio) if user.carb_ratio else 0.55
    )
    score = NutritionService.calculate_diet_score(stats['total'], targets)
    remaining = {
        'calorie': max(0, targets['calorie'] - stats['total']['calorie']),
        'protein': max(0, targets['protein'] - stats['total']['protein']),
        'carb': max(0, targets['carb'] - stats['total']['carb']),
        'fat': max(0, targets['fat'] - stats['total']['fat'])
    }
    return success({
        'date': today.isoformat(),
        'actual': stats['total'],
        'target': targets,
        'remaining': remaining,
        'by_meal_type': stats['by_meal_type'],
        'score': score,
        'bmi': float(user.bmi),
        'bmi_status': NutritionService.get_bmi_status(float(user.bmi))
    })

6.3.2  AI饮食建议与智能问答实现
AI辅助功能建立在用户身体档案和饮食统计结果之上，并不是简单调用一个通用聊天接口。系统在执行AI建议、饮食评价、下一餐推荐和智能问答之前，会先判断AI功能是否开启、用户当日剩余调用次数是否充足，然后再提取用户当天热量和三大营养素摄入情况、目标值以及健康目标等上下文信息，最后交由AIService完成模型调用。这样设计可以使AI的回答更加贴合用户当前饮食场景。
为保证系统运行可控，后端ai.py中实现了/status、/suggestion、/evaluate、/recommend-food和/chat等接口，并统一返回免责声明信息。同时，系统会把每次AI调用结果保存为AISuggestion记录，用于后续后台统计与审计。智能问答方面，前端AI页面采用对话式布局，并在本地缓存聊天记录，以增强连续交互体验。关键代码如下：
@ai_bp.route('/chat', methods=['POST'])
@login_required
def ai_chat():
    user = g.current_user
    data = request.get_json() or {}
    question = data.get('question', '').strip()
    if not question:
        return error('请输入问题')
    if len(question) > 500:
        return error('问题过长，请精简')
    if not AIService.is_enabled():
        return error('AI功能暂未开启')
    if AIService.get_remaining_calls(user.id) <= 0:

    content, err = AIService.chat(user, daily_stats, targets, question)
    if err:
        return error(err)
    return success({
        'question': question,
        'answer': content,
        'remaining_calls': AIService.get_remaining_calls(user.id),
        'disclaimer': '仅供参考，不替代专业医疗建议'
    })

6.4.3  系统配置与AI调用统计实现
后台系统配置模块主要用于维护AI相关运行参数和公告信息。该部分接口位于server/app/api/admin/config.py中，普通管理员可以查看配置内容，而超级管理员才具备修改权限。可维护内容包括AI功能开关、每日调用限额、默认模型、接口地址、提示词模板和公告信息等。这样设计使AI能力不再以硬编码形式存在，而是可以由后台统一控制与调整。
此外，系统还实现了AI调用统计接口，用于汇总当天总调用次数、各类型调用分布、Token消耗以及平均响应时间。这些结果既能够辅助管理员观察AI模块运行情况，也有助于后续论文中的系统运行分析。关键代码如下：
@admin_config_bp.route('/stats/ai', methods=['GET'])
@admin_required
def get_ai_stats():
    today = date.today()
    today_count = AISuggestion.query.filter(
        func.date(AISuggestion.created_at) == today
    ).count()
    type_stats = db.session.query(
        AISuggestion.suggestion_type,
        func.count(AISuggestion.id)
    ).filter(
        func.date(AISuggestion.created_at) == today
    ).group_by(AISuggestion.suggestion_type).all()
    today_tokens = db.session.query(
        func.sum(AISuggestion.tokens_used)
    ).filter(
        func.date(AISuggestion.created_at) == today
    ).scalar() or 0
    return success({
        'today': {
            'total_calls': today_count,
            'tokens_used': today_tokens,
            'by_type': {t: c for t, c in type_stats}
        }
    })
